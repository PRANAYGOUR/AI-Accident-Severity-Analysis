# AI Model Architecture & Training Methodology

This document explains the Machine Learning architecture, the training process, and the logic used to analyze and determine the severity of traffic accidents in this project.

## 1. Dual-Model Architecture

Instead of relying on a single AI to perform the multi-faceted task of analyzing an accident, the system leverages two specialized neural networks in tandem using the **YOLOv8** framework:

### A. Object Detection: `yolov8n.pt`
- **Purpose**: Strict vehicle counting.
- **How it works**: This is the pre-trained base YOLO object detection model. It scans the provided frame specifically looking for instances of class IDs `2` (car), `3` (motorcycle), `5` (bus), and `7` (truck). It does not know what an accident is; it only provides the system with a spatial awareness of how many vehicles are in the scene.

### B. Image Classification: `best.pt`
- **Purpose**: Crash classification and confidence evaluation.
- **How it works**: This is a custom-trained version of the YOLO classification model. It categorizes the entire frame into a specific class (`Accident` vs `Normal`) and attaches a strict mathematical confidence probability to that assumption (e.g., `89%` confident it is a crash).

## 2. Training Process (`train_classifier.py`)

To ensure the AI recognizes real-world accident conditions instead of just generic objects, we applied **Transfer Learning**.

- **The Dataset (`dataset/data`)**: Thousands of images of both severe car crashes and normal traffic flow were gathered and organized into subdirectories representing their true class.
- **The Execution**: The script `train_classifier.py` loaded the base classification model (`yolov8n-cls.pt`) and evaluated the local dataset over `10 epochs`. 
- **The Result**: At each iteration, the neural network tested itself against the labeled dataset, adjusting its internal weights to minimize loss. Upon completing the final epoch, it extracted its most accurate set of weights and saved them persistently into `ml/runs/accident_classifier/weights/best.pt`.

## 3. The Live Analysis Pipeline (`analyze.py`)

When the backend server receives media (an image or video frame), it calls the conductor script `analyze.py`, which integrates both models.

1. **Extraction**: If a video is uploaded, the script extracts 2 frames per second to ensure real-time efficiency without bogging down the CPU.
2. **Detection**: `yolov8n.pt` processes the frame, counting all distinct vehicles.
3. **Classification**: `best.pt` processes the exact same frame, evaluating the likelihood of a crash.
4. **Visual Proof Generation**: OpenCV is used to explicitly draw the bounding boxes generated by `yolov8n.pt` over the vehicles and prints the classification result (`ACCIDENT` or `NORMAL`) directly onto the frame, saving it into an annotated version (re-compiling into `.webm` for videos).
5. **Severity Scoring Logic**: 
   - If *no accident* is detected: The severity remains very low, only slightly increasing based on the number of vehicles (heavy traffic).
   - If an *accident* is detected: The score spikes violently. It starts at a base of `50%`, then adds up to `30%` based on the AI's literal confidence level of the crash, plus an additional penalty for every vehicle involved in the entanglement.
   
The script finally bundles the `vehicle_count`, `accident_detected` (boolean), `confidence` rating, `class_name`, total `severity_score`, and the path of the annotated proof media into a clean JSON output for the Node.js backend to consume and log.
